# Independent Code-Review Package (Claude Code SDK) Implementation Plan

## Overview

Add a fully independent package at `packages/code-review`: a terminal-runnable CLI that reads a **git diff from stdin**, sends it to Claude via the **Claude Code SDK** (`@anthropic-ai/claude-agent-sdk`), and produces a structured code review. The review is obtained by forcing the model to call a single mandatory **`review` tool** whose input is a **Zod** schema (so requirement #6 — "the response should be the valid call to `review`" — is satisfied literally: the model's output _is_ a validated call to `review`). The validated review is then rendered to the terminal (human-readable by default, raw JSON with `--json`).

This is an initial version: diff in → review out, run locally. No CI wiring, no PR integration, no multi-file orchestration.

## Current State Analysis

- **Package manager:** pnpm `11.6.0`, pinned in both `package.json` (`packageManager`) and `mise.toml` (`npm:pnpm = "11.6.0"`). Node `24` via `mise.toml`. The Claude Agent SDK requires Node ≥18, so the toolchain is compatible.
- **Workspace:** `pnpm-workspace.yaml` exists but carries only `overrides` and `allowBuilds` — there is **no `packages:` glob**, and there is **no `packages/` directory**. The repo is therefore not yet a monorepo; adding a workspace member is a deliberate, additive change to `pnpm-workspace.yaml`.
- **LLM dependency direction:** the app talks to models via `@openrouter/sdk` / `@openrouter/agent` today — **not** the Anthropic SDK. The Claude Code SDK is a brand-new dependency for this repo.
- **Existing libs available:** `zod@^4` (so `z.toJSONSchema` and the modern API are available), `effect@^3`, `ts-pattern@^5`. The new package will **not** depend on `effect` — it stays minimal plain TypeScript (decision below).
- **Conventions scope:** the hexagonal/Effect conventions in `CLAUDE.md` and `docs/reference/conventions/` are scoped to the Astro app under `src/`. The independent package is not bound by them.
- **Lint/format:** root `lint` is `eslint .` with type-checked rules tuned for the Astro app (path alias `@/*`, project tsconfig); root `format` is `prettier --write .`. Lefthook lints **staged files** pre-commit (per project memory). A new package under `packages/**` must not be dragged into the app's type-checked lint program or it will fail; it must be excluded from the root ESLint scope.

## Desired End State

From a clean checkout, after one `pnpm install` at the repo root:

```
git diff | pnpm --filter code-review review            # pretty review to stdout
git diff | pnpm --filter code-review review -- --json   # raw validated JSON
git diff | pnpm --filter code-review review -- --model claude-opus-4-8
```

…produces a Zod-validated review (`summary` + `findings[]` + `verdict`) generated by Claude through the `review` tool call. `ANTHROPIC_API_KEY` in the environment is the only runtime credential. Typecheck and the package's unit tests pass; the root app build and lint are unaffected.

### Key Discoveries:

- **Claude Code SDK = `@anthropic-ai/claude-agent-sdk`** (renamed from `@anthropic-ai/claude-code`); TypeScript, ESM, Node ≥18; verified at SDK `v0.3.195`.
- **Custom-tool contract** (verified against the SDK custom-tools + TypeScript reference docs):
  - `import { tool, createSdkMcpServer } from "@anthropic-ai/claude-agent-sdk"`.
  - `tool(name, description, zodShape, handler)` — the SDK **validates input against the Zod shape before calling the handler** (so the handler receives already-validated args; on mismatch the SDK re-prompts the model). This is where requirement #4 (Zod for structured I/O) is satisfied natively.
  - `createSdkMcpServer({ name, version, tools: [reviewTool] })`; passed to `query()` via `options.mcpServers` as an **object map** `{ "code-review": server }`.
  - The MCP tool name for `allowedTools` is `mcp__<serverName>__<toolName>` → **`mcp__code-review__review`** (hyphen in the server name is fine — the docs use it directly).
- **Forcing a single call** (no Messages-API `tool_choice` exists in the Agent SDK): `tools: []` (drops all built-ins from context) + `allowedTools: ["mcp__code-review__review"]` + `permissionMode: "bypassPermissions"` + `maxTurns: 1` + a system prompt instructing the model to call `review` exactly once.
- **Capture + result detection:** the handler closes over an outer variable to capture the review; the terminal `result` message (`message.type === "result"`, `subtype === "success"`) ends the loop. If `capturedReview` is still undefined at that point, the model never reviewed → throw.
- **Auth:** `ANTHROPIC_API_KEY` env var (headless); no constructor arg required.
- **Node built-ins suffice for the CLI:** `node:util` `parseArgs` for flags, stdin read via `process.stdin` — no `commander`/`yargs` dependency needed.

## What We're NOT Doing

- No CI/GitHub Action, no PR-comment posting, no git hook integration.
- The tool does **not** run `git` itself — the diff arrives on stdin.
- No multi-file fan-out, chunking of huge diffs, retry/repair loops, or caching.
- No native `outputFormat` structured-output mode and no "ask for JSON then parse" path — the `review` tool call is the single sanctioned mechanism.
- The package does **not** adopt the app's Effect/hexagonal conventions, and is **not** added to the root type-checked ESLint program.
- No compiled `dist`/published binary — it runs via `tsx`.
- Exit code is not derived from `verdict` (e.g. non-zero on `request_changes`) in this version — success exits 0 regardless of verdict.

## Implementation Approach

Four phases, each independently verifiable. Phase 1 makes the package exist and install as a workspace member. Phase 2 defines the framework-free Zod contracts (input + output) and the prompt — the shared spine both the engine and CLI depend on. Phase 3 builds the SDK engine around the mandatory `review` tool. Phase 4 wires the terminal entry point (stdin → engine → rendered output) and error handling, delivering the end-to-end local run.

Dependency keeps minimal: runtime deps are `@anthropic-ai/claude-agent-sdk` + `zod`; dev deps are `tsx`, `typescript`, `@types/node`, `vitest`.

## Critical Implementation Details

- **Workspace member, but isolated from app tooling.** Adding `packages/*` to `pnpm-workspace.yaml`'s (new) `packages:` key brings the package under one `pnpm install`. But the root `eslint .` runs the Astro app's type-checked config; the new package must be added to the root ESLint `ignores` (it carries its own minimal setup / Prettier formatting), or `eslint .` and the Lefthook pre-commit hook will fail on it.
- **SDK build-script allowlist.** `@anthropic-ai/claude-agent-sdk` may ship an install/build step (it bundles/launches the `claude` runtime). pnpm 11 blocks unlisted build scripts; if install warns about an ignored build script for the SDK (or a native sub-dependency), add it to the `allowBuilds` map in `pnpm-workspace.yaml` (same pattern as the existing `@openrouter/sdk: true` entry).
- **Forcing exactly one tool call is a combination, not a flag.** The Agent SDK has no `tool_choice: "required"`. The reliable single-shot recipe is `tools: []` + `allowedTools: ["mcp__code-review__review"]` + `permissionMode: "bypassPermissions"` + `maxTurns: 1` + an explicit system-prompt instruction. The "model never called the tool" case must still be handled (captured value undefined at the `result` message → error).
- **`tsc --noEmit` deprecation guard.** Per project memory, plain `tsc --noEmit` can abort on a tsconfig deprecation under the installed TypeScript 6; if the package's typecheck hits this, pass `--ignoreDeprecations 6.0`.

## Phase 1: Scaffold package + workspace wiring

### Overview

Create the package skeleton and make it install and run as a pnpm workspace member, without disturbing the root app's build/lint.

### Changes Required:

#### 1. Register the workspace glob

**File**: `pnpm-workspace.yaml`

**Intent**: Make `packages/*` part of the pnpm workspace so the new package installs under the existing root `pnpm install` and resolves via `pnpm --filter`.

**Contract**: Add a top-level `packages:` key listing `packages/*`, alongside the existing `overrides` / `allowBuilds` keys (additive — leave those intact). If the SDK install emits an ignored-build-script warning, add the offending package to `allowBuilds`.

#### 2. Package manifest

**File**: `packages/code-review/package.json`

**Intent**: Declare the independent package, its `tsx`-based run script, runtime deps (`@anthropic-ai/claude-agent-sdk`, `zod`), and dev deps (`tsx`, `typescript`, `@types/node`, `vitest`).

**Contract**: `name: "code-review"` (or scoped, e.g. `@snapchef/code-review`), `"type": "module"`, `"private": true`, `engines.node >= 18`. Scripts: `review` → `tsx src/cli.ts`, `test` → `vitest run`, `typecheck` → `tsc --noEmit`. The `review` script is the documented terminal entry (`pnpm --filter code-review review`). A `bin` entry is optional and out of scope for v0.

#### 3. Package TypeScript config

**File**: `packages/code-review/tsconfig.json`

**Intent**: Strict, modern ESM config independent of the root Astro tsconfig.

**Contract**: `module`/`moduleResolution` `NodeNext`, `target` `ES2022`, `strict: true`, `types: ["node"]`, `include: ["src"]`. Does **not** extend the root `astro/tsconfigs/strict` config (no React/Astro concerns here).

#### 4. Exclude the package from the root lint program

**File**: `eslint.config.js`

**Intent**: Keep the independent package out of the Astro app's type-checked ESLint run so `eslint .` and the Lefthook pre-commit hook stay green.

**Contract**: Add `packages/**` to the root flat-config global `ignores`. (Prettier may still format the package; that's harmless.)

## Phase 2: Domain layer — Zod contracts + prompt

### Overview

Define the framework-free Zod schemas for the review **output** and the CLI **input/options**, plus the prompt that instructs Claude — the shared contract Phases 3 and 4 both consume.

### Changes Required:

#### 1. Review output schema

**File**: `packages/code-review/src/review.ts`

**Intent**: The Zod schema (and inferred type, same-name) describing the structured review the model must produce. This is the input shape of the `review` tool and the object the CLI renders.

**Contract**: `Review = { summary: string; findings: Finding[]; verdict: "approve" | "comment" | "request_changes" }` where `Finding = { severity: "critical" | "major" | "minor" | "nit"; file: string; line?: number; title: string; detail: string; suggestion?: string }`. Export both the schema and the inferred type under the same name (project Zod convention). The schema's _shape_ (the raw object passed to `tool()`) must be exported/derivable, since `tool()` takes a Zod raw shape, not a wrapped `z.object`.

#### 2. CLI options schema

**File**: `packages/code-review/src/options.ts`

**Intent**: Validate parsed CLI flags so bad input fails clearly and `model` has a default.

**Contract**: `CliOptions = { json: boolean (default false); model: string (default "claude-sonnet-4-6") }`. Parsed from `node:util` `parseArgs` output in Phase 4, then `CliOptions.parse(...)`.

#### 3. Prompt builder

**File**: `packages/code-review/src/prompt.ts`

**Intent**: Produce the system prompt (reviewer role + "you MUST call the `review` tool exactly once; do not reply with prose") and the user prompt embedding the diff.

**Contract**: Two pure functions/strings — a constant `SYSTEM_PROMPT` and `buildUserPrompt(diff: string): string`. The system prompt instructs severity usage and the single mandatory `review` call; the user prompt wraps the raw diff.

### Success Criteria:

#### Automated Verification:

- `pnpm --filter code-review typecheck` passes
- Unit test: `Review.parse(validFixture)` succeeds and `Review.parse(invalidFixture)` throws (`vitest run`)
- Unit test: `CliOptions.parse({})` yields the documented defaults (`json: false`, `model: "claude-sonnet-4-6"`)

#### Manual Verification:

- Review schema fields match the decision (summary + findings[severity/file/line?/title/detail/suggestion?] + verdict) and read sensibly for a real diff

## Phase 3: Claude Code SDK engine — the `review` tool

### Overview

Build the engine that turns a diff string into a validated `Review` by forcing Claude to call the `review` tool, in a single-shot headless run with all built-in tools disabled.

### Changes Required:

#### 1. Review engine

**File**: `packages/code-review/src/engine.ts`

**Intent**: Define the mandatory `review` tool from the Zod shape, run `query()` constrained to a single tool call with no filesystem/bash access, capture the validated args in the handler, and return the `Review` (or throw if the model never called the tool / the run errored).

**Contract**: Exposes `runReview(diff: string, opts: { model: string }): Promise<Review>`. Wiring (verified against `@anthropic-ai/claude-agent-sdk` v0.3.195 — confirm names against the installed version):

```ts
import { tool, createSdkMcpServer, query } from "@anthropic-ai/claude-agent-sdk";

let captured: Review | undefined;
const reviewTool = tool(
  "review",
  "Submit the completed code review.",
  ReviewShape, // the raw Zod shape from review.ts
  async (args) => {
    // args already Zod-validated by the SDK
    captured = args;
    return { content: [{ type: "text", text: "Review recorded." }] };
  },
);
const server = createSdkMcpServer({ name: "code-review", version: "0.0.1", tools: [reviewTool] });

for await (const message of query({
  prompt: buildUserPrompt(diff),
  options: {
    model: opts.model,
    maxTurns: 1,
    tools: [], // disable all built-ins
    allowedTools: ["mcp__code-review__review"], // only the review tool
    permissionMode: "bypassPermissions", // headless, no prompts
    systemPrompt: { type: "custom", prompt: SYSTEM_PROMPT },
    mcpServers: { "code-review": server }, // object map, key = server name
  },
})) {
  if (message.type === "result") {
    if (message.subtype !== "success") throw new Error(/* surface message.errors */);
    if (!captured) throw new Error("Model did not call the review tool");
  }
}
return captured!;
```

The snippet is included because the SDK surface is non-obvious and load-bearing (the `tool()` signature, the `mcpServers` object-map shape, the `mcp__code-review__review` allow-string, and the `tools: []` + `allowedTools` lockdown are the crux of the whole change). Keep `runReview` free of CLI/stdout concerns.

### Success Criteria:

#### Automated Verification:

- `pnpm --filter code-review typecheck` passes (SDK types resolve; `runReview` signature is correct)
- Unit test: with the SDK module mocked, the `review` tool handler captures args and `runReview` resolves to that `Review`; a `result` with no captured value rejects with the "did not call" error

#### Manual Verification:

- With a real `ANTHROPIC_API_KEY`, calling `runReview(sampleDiff, { model: "claude-sonnet-4-6" })` returns a sensible, schema-valid review for a known diff
- Overriding `model` (e.g. `claude-opus-4-8`) is honored

## Phase 4: CLI entry + output rendering

### Overview

Wire the terminal entry point: read the diff from stdin, parse flags, invoke the engine, and render the review (pretty by default, JSON with `--json`), with clear errors.

### Changes Required:

#### 1. Output renderer

**File**: `packages/code-review/src/render.ts`

**Intent**: Turn a `Review` into terminal output — a readable, grouped summary by default; the raw validated object as pretty JSON when `--json` is set.

**Contract**: `renderReview(review: Review, opts: { json: boolean }): string`. Pretty path: a header line with `verdict`, the `summary`, then findings grouped/sorted by `severity` showing `file:line — title`, `detail`, and `suggestion` when present. JSON path: `JSON.stringify(review, null, 2)`. Pure function (returns a string; the CLI prints it) so it is unit-testable.

#### 2. CLI entry

**File**: `packages/code-review/src/cli.ts`

**Intent**: The `tsx`-run entry: read stdin to a string, parse args via `node:util` `parseArgs`, validate with `CliOptions`, call `runReview`, print `renderReview`, and map failures to clear messages + non-zero exit.

**Contract**: Reads the full diff from `process.stdin`. Flags: `--json` (boolean), `--model <id>` (string). Error handling with distinct messages and non-zero exit for: empty/no stdin ("no diff provided on stdin"), missing `ANTHROPIC_API_KEY`, engine failure (run errored or model never called `review`). Success prints the rendered review and exits 0.

### Success Criteria:

#### Automated Verification:

- `pnpm --filter code-review typecheck` passes
- Unit test: `renderReview(fixture, { json: true })` is valid JSON that round-trips to the fixture; `renderReview(fixture, { json: false })` contains the verdict, summary, and each finding's `file` and `title`
- Unit test / harness: invoking the CLI with empty stdin exits non-zero with the "no diff" message (engine not called)

#### Manual Verification:

- `git diff | pnpm --filter code-review review` prints a readable review for real local changes
- `git diff | pnpm --filter code-review review -- --json` prints valid JSON matching the `Review` schema
- `-- --model claude-opus-4-8` is honored; running with `ANTHROPIC_API_KEY` unset prints the credential error and exits non-zero

## Testing Strategy

### Unit Tests (vitest, in the package):

- `Review` schema accepts a valid fixture and rejects malformed ones (missing `verdict`, bad `severity`).
- `CliOptions` defaults (`json: false`, `model: "claude-sonnet-4-6"`).
- `renderReview` pretty + JSON outputs (pure function — no SDK/network).
- `runReview` with the SDK module mocked: handler captures args → resolved `Review`; no capture at `result` → rejects.

### Integration / Manual Testing:

- End-to-end with a real `ANTHROPIC_API_KEY` against a small known diff (e.g. a deliberately buggy change) to confirm the model calls `review` and findings are sensible.
- `--json` vs pretty output, `--model` override, empty-stdin and missing-key error paths.

### Manual Testing Steps:

1. `pnpm install` at root; confirm `pnpm lint` and `pnpm build` still pass.
2. Export `ANTHROPIC_API_KEY`. Make a small code change, then `git diff | pnpm --filter code-review review`.
3. Re-run with `-- --json` and with `-- --model claude-opus-4-8`.
4. Run with no stdin and with the key unset; confirm clear errors + non-zero exit.

## Performance Considerations

Single LLM round-trip per invocation; latency is dominated by the model. Sonnet 4.6 default keeps cost/latency modest for a v0; Opus 4.8 available via `--model` for deeper review. Very large diffs may approach context limits — out of scope for this version (no chunking).

## Migration Notes

Purely additive: a new `packages/` tree, a new `packages:` glob in `pnpm-workspace.yaml`, and a `packages/**` entry in the root ESLint `ignores`. No changes to the Astro app, its build, or its deploy. No database or Supabase involvement.
