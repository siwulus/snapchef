# Extensible Coding-Style Conventions for Agentic Coding Implementation Plan

## Overview

Snapchef needs a single, extensible home for coding-style conventions that AI
agents reliably read and obey. Today, style guidance is scattered: `CLAUDE.md`
covers project rules but not micro-conventions (naming, paradigm), and
`docs/reference/forms.md` teaches Zod patterns that will conflict with the new
rules. This plan creates a **conventions registry** under
`docs/reference/conventions/` — one markdown file per style domain plus an
index — wired into `CLAUDE.md` via `@`-import and a binding "Coding Style"
pointer. It seeds the registry with two domains (generic style, Zod naming),
reconciles `forms.md`, and refactors existing source into compliance.

The defining property is **extensibility**: adding a new style domain later is
"create one file + add one line to the index" — nothing else.

## Current State Analysis

- **`CLAUDE.md` is the single agent entry point.** `AGENTS.md → CLAUDE.md` and
  `.agents → .claude` symlinks mean one file serves Claude Code, Cursor, and
  other AGENTS-aware tools. `CLAUDE.md` already uses `@path` import-style
  references (`@package.json`, `@eslint.config.js`, `@astro.config.mjs`) —
  Claude Code auto-pulls those into context, and resolves nested `@`-imports in
  imported files.
- **`docs/reference/` is the established home for detailed reference docs**
  linked from `CLAUDE.md`. `docs/reference/forms.md` exists;
  `docs/reference/contract-surfaces.md` is referenced by `CLAUDE.md` but does
  not yet exist (out of scope here). The conventions registry slots into this
  same pattern.
- **Zod naming conflict.** Existing code uses _different_ names for schema vs
  type: `src/lib/validation/auth.ts` exports `signInSchema` + `type SignInInput`
  (and `signUpSchema`/`SignUpInput`, plus `signUpServerSchema`). `forms.md`
  teaches the same `recipeSchema`/`RecipeInput` split. The new convention wants
  the **same name** for schema and type (`export const SignIn = z.object(...)` +
  `export type SignIn = z.infer<typeof SignIn>`). TypeScript allows a `const` and
  a `type` to share an identifier (separate value/type namespaces), so this is
  mechanically clean.
- **Arrow-function conflict.** 33 `function` occurrences exist across 17 `src`
  files. **5 are vendored shadcn `src/components/ui/*`** (`card`, `label`,
  `button`, `form`, `input`) — regenerated by `npx shadcn add`, so refactoring
  them is futile (overwritten on regen). The remaining ~12 files (auth
  components, `useZodForm`, `submitJson`, `utils`, `supabase`, the two auth API
  routes) are valid refactor targets.
- **A `/10x-rule-review` skill** scores rule-for-AI files on agent-obedience —
  the registry should be authored to that bar (imperative phrasing + examples,
  critical instruction near the top).

### Key Discoveries:

- `CLAUDE.md:1-13` — "Hard Rules" block is where binding, top-weighted
  instructions live; the Coding Style pointer belongs here or immediately after.
- `src/lib/validation/auth.ts:3-25` — the only Zod definition file; the full
  rename surface.
- Zod import sites: `src/components/auth/SignInForm.tsx:11-12,19,25`,
  `src/components/auth/SignUpForm.tsx:11-12,22,34`,
  `src/pages/api/auth/signin.ts:3,34`, `src/pages/api/auth/signup.ts:3,34`.
- `docs/reference/forms.md:13-18,98-112` — teaches the old `recipeSchema` /
  `RecipeInput` split; must be updated to the same-name pattern.
- Vendored (exclude): `src/components/ui/{card,label,button,form,input}.tsx`.

## Desired End State

- `docs/reference/conventions/` exists with `README.md` (index/registry),
  `generic.md`, and `zod.md`.
- `CLAUDE.md` has a "Coding Style" section that `@`-imports the registry index
  and binds agents to obey it ("consult and obey before writing code").
- The registry README documents the "how to add a new convention domain" recipe
  so a future domain is a one-file-plus-one-line addition.
- `docs/reference/forms.md` teaches the same-name Zod convention and arrow style
  — no doc contradicts a rule.
- Existing source complies: `auth.ts` and its 4 import sites use same-name
  schema+type; non-vendored `src` files use arrow functions where applicable.
- **Verification:** `npm run lint` and `npx tsc`/typecheck pass; grep confirms no
  `Schema`/`Input` split remains in `auth.ts`; grep confirms no `function`
  keyword in non-vendored refactored files (allowing documented exceptions); the
  registry index resolves via `@`-import.

## What We're NOT Doing

- Not creating `docs/reference/contract-surfaces.md` (separate, pre-existing gap).
- Not refactoring vendored `src/components/ui/*` shadcn primitives.
- Not adding lint rules / ESLint config to _enforce_ arrow or functional style
  programmatically — rules are documented + agent-obeyed; automated enforcement
  is a possible future domain-specific follow-up, explicitly out of scope.
- Not authoring style domains beyond `generic` and `zod` (e.g. React, Astro,
  testing) — the mechanism makes those trivial to add later, but they are not
  part of this change.
- Not changing runtime behavior — all migrations are rename/syntax refactors that
  preserve behavior.

## Implementation Approach

Build the mechanism first (Phase 1) so the rules physically exist and are
agent-discoverable before anything is migrated against them. Then remove the
contradiction in the docs (Phase 2). Then migrate existing code (Phase 3) so the
codebase itself is a compliant reference example. Each phase is independently
verifiable (lint/typecheck) and committable.

The registry uses **recursive `@`-import**: `CLAUDE.md` `@`-imports
`docs/reference/conventions/README.md`, which in turn `@`-imports each domain
file. This guarantees all active conventions load into agent context while
keeping the "add a domain = add a file + one `@` line" extensibility property.
Each rule is written as an **imperative statement + a tiny `✓ good` / `✗ bad`
snippet** — the format that maximizes agent compliance and scores well on
`/10x-rule-review`.

## Critical Implementation Details

- **Recursive `@`-import depth.** Claude Code resolves `@`-imports inside
  imported files. The registry relies on this: `CLAUDE.md → README.md →
{generic,zod}.md`. Keep the chain shallow (2 hops) so it loads reliably; the
  Coding Style pointer in `CLAUDE.md` must _also_ name the directory in prose, so
  agents that don't follow imports still know to read it.
- **Zod same-name in TypeScript.** `export const SignIn = z.object(...)` and
  `export type SignIn = z.infer<typeof SignIn>` coexist because TS keeps value
  and type namespaces separate. The `typeof SignIn` in the type refers to the
  const — this is the canonical Zod idiom and must be shown verbatim in `zod.md`.
- **`signUpSchema` is built via `.refine()` on an internal `signUpBase`.** When
  renaming, the public schema becomes `SignUp`; the internal base can become
  `SignUpBase` (unexported). `signUpServerSchema` → `SignUpServer`. Preserve the
  refine logic and the `path: ["confirmPassword"]` exactly.

## Phase 1: Conventions Registry + CLAUDE.md Wiring

### Overview

Create the extensible registry directory, seed it with the index and two domain
files, and bind it into `CLAUDE.md` so agents load and obey it.

### Changes Required:

#### 1. Registry index / registry doc

**File**: `docs/reference/conventions/README.md`

**Intent**: The entry point and registry for all coding-style domains. Explains
that these conventions are binding for agents, lists each domain file with a
one-line summary, recursively `@`-imports each domain file so all active rules
load into context, and documents the "How to add a new convention domain"
recipe that makes the mechanism extensible.

**Contract**: Must contain (a) a short binding preamble, (b) a registry
table/list with one row per domain (`generic`, `zod`) linking to its file, (c)
recursive imports `@./generic.md` and `@./zod.md`, and (d) a "How to add a new
convention" section specifying the file template and the single index line to
add. Adding a domain later must require editing only this file (one row + one
`@` line) plus creating the new domain file.

#### 2. Generic style domain

**File**: `docs/reference/conventions/generic.md`

**Intent**: Seed the generic-style rules: strongly prefer arrow functions over
the `function` keyword (with documented pragmatic exceptions), and strongly
prefer functional-style programming (`map`/`flatMap`/`reduce`/etc.) over
imperative `for`/`forEach` loops and over `if`-chains where a functional mapping
expresses the same logic.

**Contract**: Each rule = one imperative line + a tiny `✓ good` / `✗ bad` code
snippet. The arrow rule must include an "Exceptions" note (e.g. cases where
hoisting or a framework idiom legitimately needs `function`, and the vendored
`src/components/ui/*` exclusion). Rules phrased as binding ("Use…", "Prefer…",
not "Consider…").

#### 3. Zod naming domain

**File**: `docs/reference/conventions/zod.md`

**Intent**: Seed the Zod naming convention: schema and inferred type share the
same name, following `export const SomeType = z.object(...)` +
`export type SomeType = z.infer<typeof SomeType>`.

**Contract**: Imperative rule + a `✓ good` / `✗ bad` snippet showing the
same-name pattern vs the old `Schema`/`Input` split. Snippet must show the
`z.infer<typeof SomeType>` self-reference verbatim (the non-obvious TS idiom).

#### 4. CLAUDE.md Coding Style pointer

**File**: `CLAUDE.md`

**Intent**: Add a binding "Coding Style" section (under or adjacent to "Hard
Rules", high in the file) that `@`-imports the registry index and instructs
agents in imperative prose to consult and obey the conventions in
`docs/reference/conventions/` before writing code.

**Contract**: New section containing the prose directive naming the directory +
`@docs/reference/conventions/README.md`. Must not duplicate individual rule text
(single source of truth stays in the domain files).

### Success Criteria:

#### Automated Verification:

- Registry files exist: `ls docs/reference/conventions/{README.md,generic.md,zod.md}`
- `CLAUDE.md` references the registry: `grep -q "docs/reference/conventions" CLAUDE.md`
- Markdown formatting passes: `npm run format` (or `npx prettier --check` on the new files)
- Lint passes: `npm run lint`

#### Manual Verification:

- Adding a hypothetical new domain is demonstrably one-file-plus-one-line (walk the README recipe).
- A fresh agent reading `CLAUDE.md` would load the conventions via `@`-import and treat them as binding.
- Rules are skimmable: imperative line + example, no buried prose.

**Implementation Note**: After completing this phase and all automated
verification passes, pause for manual confirmation that the registry reads
clearly and the extensibility recipe is sound before proceeding.

## Phase 3: Migrate Existing Code

### Overview

Bring existing source into compliance: same-name Zod rename in `auth.ts` and its
import sites, and arrow-function refactor across non-vendored `src` files.
Vendored `src/components/ui/*` is excluded.

### Changes Required:

#### 1. Zod same-name rename

**File**: `src/lib/validation/auth.ts`

**Intent**: Rename the schema+type pairs to the same-name convention:
`signInSchema`/`SignInInput` → `SignIn`; `signUpSchema`/`SignUpInput` → `SignUp`
(internal `signUpBase` → `SignUpBase`); `signUpServerSchema` → `SignUpServer`.
Preserve all validation logic, messages, and the `.refine()` path.

**Contract**: Exports become `export const SignIn`, `export type SignIn`,
`export const SignUp`, `export type SignUp`, `export const SignUpServer`. The
`.refine()` on `SignUpBase` and `path: ["confirmPassword"]` are unchanged.

#### 2. Zod import sites

**File**: `src/components/auth/SignInForm.tsx`, `src/components/auth/SignUpForm.tsx`, `src/pages/api/auth/signin.ts`, `src/pages/api/auth/signup.ts`

**Intent**: Update imports and usages to the new identifiers
(`SignIn`/`SignUp`/`SignUpServer`), replacing both the schema value imports and
the `SignInInput`/`SignUpInput` type imports with the same-name type.

**Contract**: Import sites at `SignInForm.tsx:11-12,19,25`,
`SignUpForm.tsx:11-12,22,34`, `signin.ts:3,34`, `signup.ts:3,34`. After the
change, `grep -rn "Input\b" src/components/auth src/pages/api/auth` shows no
`SignInInput`/`SignUpInput` references. (The `onSubmit` here is also an arrow
target — see change #3.)

#### 3. Arrow-function refactor (non-vendored)

**File**: non-vendored `src` files using `function` — `src/components/auth/{SubmitButton,SignInForm,SignUpForm,PasswordToggle,ServerError,IconField}.tsx`, `src/components/hooks/useZodForm.ts`, `src/lib/{submitJson,utils,supabase}.ts`, `src/pages/api/auth/{signin,signup}.ts`

**Intent**: Convert `function` declarations/expressions to arrow functions where
applicable per `generic.md`, applying documented exceptions where a `function`
form is genuinely required (note each exception inline if kept).

**Contract**: **Exclude** `src/components/ui/{card,label,button,form,input}.tsx`
(vendored shadcn — regenerated by CLI). Behavior unchanged; default-export
React components converted to `const X = () => …; export default X` only where it
doesn't break Astro/React island conventions (otherwise documented as an
exception). No change to function signatures or props.

### Success Criteria:

#### Automated Verification:

- Typecheck passes (rename has no dangling refs): `npx astro check` / `npx tsc --noEmit`
- Lint passes: `npm run lint`
- No old Zod naming remains: `grep -rnE "signInSchema|signUpSchema|signUpServerSchema|SignInInput|SignUpInput" src` returns nothing
- No `function` keyword in refactored non-vendored files (excluding documented exceptions): `grep -rn "\bfunction\b" src --include=*.ts --include=*.tsx | grep -v "components/ui/"`
- Build passes: `npm run build`

#### Manual Verification:

- Sign-in and sign-up flows work end-to-end via the UI (forms submit, validation messages show, redirects fire).
- No regression in form field-error mapping (`form.setError` still matches field names).
- Refactored arrow functions read naturally; any retained `function` has a documented reason.

**Implementation Note**: After automated verification, pause for manual
confirmation that the auth flows work in the browser before considering the
change complete.

## Testing Strategy

### Unit Tests:

- No unit-test suite is established for these modules currently; rely on
  typecheck + lint + build as the automated safety net for rename/syntax refactors.

### Integration Tests:

- None automated. Manual end-to-end of auth flows substitutes.

### Manual Testing Steps:

1. Run `npm run dev`, open the sign-in page, submit valid + invalid credentials, confirm validation + redirect behavior.
2. Repeat for sign-up, including the password-mismatch case (confirms `.refine()` survived the rename).
3. Read `CLAUDE.md` top-to-bottom as if onboarding — confirm the Coding Style section makes the conventions feel binding and points to the registry.
4. Walk the "add a new domain" recipe in the registry README to confirm extensibility.

## Performance Considerations

None. All changes are documentation and behavior-preserving renames/syntax
refactors. The only context-budget consideration is the `@`-imported registry
size — keep domain files terse (imperative + tiny snippet) to limit token cost.

## Migration Notes

The Zod rename is a breaking identifier change but fully internal (no external
API). All call sites are updated in the same phase, so there is no
backward-compat window to manage. No database or Worker-version concerns apply.
