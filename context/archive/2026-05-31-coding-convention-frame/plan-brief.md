# Extensible Coding-Style Conventions for Agentic Coding — Plan Brief

> Full plan: `context/changes/coding-convention-frame/plan.md`

## What & Why

Snapchef has no single, extensible home for micro-level coding conventions
(naming, paradigm) that AI agents reliably read and obey. This change creates a
**conventions registry** under `docs/reference/conventions/` — one file per
style domain plus an index — wired into `CLAUDE.md` so agents auto-load it and
treat it as binding. It seeds two domains (generic style, Zod naming) and the
defining property is extensibility: adding a future domain is "create one file +
add one index line."

## Starting Point

`CLAUDE.md` is the single agent entry point (with `AGENTS.md`/`.agents`
symlinks) and already uses recursive `@`-imports. `docs/reference/` is the
established home for detailed docs (`forms.md` exists). No conventions registry
exists yet, and `forms.md` plus `src/lib/validation/auth.ts` currently use a Zod
naming pattern that contradicts the new rule.

## Desired End State

`docs/reference/conventions/{README,generic,zod}.md` exist and are `@`-imported

- pointed to by a binding "Coding Style" section in `CLAUDE.md`. `forms.md` no
  longer contradicts the rules. Existing source complies: `auth.ts` uses same-name
  schema+type, and non-vendored `src` files use arrow functions.

## Key Decisions Made

| Decision                | Choice                                         | Why (1 sentence)                                                         |
| ----------------------- | ---------------------------------------------- | ------------------------------------------------------------------------ |
| Where rules live        | Dedicated `docs/reference/conventions/` files  | Matches existing `forms.md` pattern; keeps `CLAUDE.md` lean.             |
| Extensibility structure | One file per domain + index/registry           | Adding a style = one file + one index line.                              |
| Agent discovery         | `@`-import + binding pointer in `CLAUDE.md`    | Auto-loads into context _and_ gives an explicit obedience instruction.   |
| Zod naming conflict     | Same-name schema+type; refactor existing too   | User wants `SomeType = z.object` / `type SomeType = z.infer` everywhere. |
| Arrow functions         | Strong-prefer for new code + refactor existing | Enforce where possible; pragmatic documented exceptions.                 |
| Vendored shadcn `ui/*`  | Excluded from arrow refactor                   | Regenerated by `npx shadcn add` — edits would be overwritten.            |
| Rule format             | Imperative line + tiny `✓`/`✗` example         | Examples maximize agent compliance; scores well on `/10x-rule-review`.   |

## Scope

**In scope:**

- Conventions registry (`README.md` index + `generic.md` + `zod.md`).
- `CLAUDE.md` Coding Style section + `@`-import.
- Reconcile `docs/reference/forms.md`.
- Migrate `auth.ts` + 4 Zod import sites to same-name naming.
- Arrow-function refactor of non-vendored `src` files.

**Out of scope:**

- `docs/reference/contract-surfaces.md` (separate gap).
- Vendored `src/components/ui/*`.
- ESLint/automated enforcement of the rules.
- Additional style domains (React, Astro, testing).

## Architecture / Approach

`CLAUDE.md` → `@`-imports `conventions/README.md` (registry) → recursively
`@`-imports each domain file. The README also carries the "how to add a domain"
recipe, making the mechanism self-documenting and trivially extensible. Each
rule is an imperative statement + a minimal good/bad snippet. Build mechanism
first, reconcile docs, then migrate code so the codebase becomes a compliant
example.

## Phases at a Glance

| Phase                          | What it delivers                                         | Key risk                                                                             |
| ------------------------------ | -------------------------------------------------------- | ------------------------------------------------------------------------------------ |
| 1. Registry + CLAUDE.md wiring | Extensible registry + seed rules + binding agent pointer | Recursive `@`-import not loading — mitigated by prose pointer naming the dir.        |
| 2. Reconcile reference docs    | `forms.md` examples match new conventions                | Missing an example region (low — 3 known regions).                                   |
| 3. Migrate existing code       | Same-name Zod rename + arrow refactor (non-vendored)     | Refactoring vendored `ui/*` (mitigated by explicit exclusion); auth-flow regression. |

**Prerequisites:** none — all inputs exist in the repo.
**Estimated effort:** ~1–2 sessions across 3 phases (small, contained diff).

## Open Risks & Assumptions

- Assumes Claude Code resolves 2-hop recursive `@`-imports reliably; mitigated by
  also naming the directory in the `CLAUDE.md` prose pointer.
- Arrow refactor of default-export React components may hit framework-idiom
  exceptions — handled by documenting exceptions rather than forcing arrows.
- No automated test suite for auth; manual end-to-end is the regression safety net.

## Success Criteria (Summary)

- A new agent reading `CLAUDE.md` loads the conventions and treats them as binding.
- Adding a new style domain later requires only a new file + one index line.
- `lint`, typecheck, and `build` pass; no old Zod naming or stray `function` in
  refactored non-vendored files; auth flows still work.
